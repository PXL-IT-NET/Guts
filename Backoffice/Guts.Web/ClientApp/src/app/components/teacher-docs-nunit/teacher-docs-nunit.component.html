Write monitored tests with NUnit

Use attributes from Guts.Client.NUnit so fixture and test runs are captured and submitted to the GUTS backoffice.

Required setup
  • Add the Guts.Client.NUnit package to your test project.

Class-level attributes

[ExerciseTestClass]

Use this attribute on a test class for tests of individual exercises within a chapter:

[ExerciseTestClass(courseCode, chapterCode, exerciseCode, optionalSourcePaths)]

[ProjectComponentTestClass]

Use this attribute on a test class for tests of project components in team-based project work.:

[ProjectComponentTestClass(courseCode, projectCode, componentCode, optionalSourcePaths)]

Parameters of the class-level attributes

  • courseCode: the code of the course in the GUTS system (can be found next to the course title).
  • chapterCode: the code of the chapter. You can choose this code. If a lector runs the tests and results are sent to https://guts-api.pxl.be/, a chapter with this code is created if it does not exist yet.
  • exerciseCode: the code of the exercise. You can choose this code. If a lector runs the tests and results are sent to https://guts-api.pxl.be/, an exercise with this code is created in the chapter if it does not exist yet.
  • componentCode: the code of the component. You can choose this code. If a lector runs the tests and results are sent to https://guts-api.pxl.be/, a component with this code is created if it does not exist yet.
  • optionalSourcePaths: optional semicolon-separated paths to source files relevant for the test. Paths are relative to the solution directory. File contents are sent to the GUTS backoffice.
Code format rule: keep code parameters lowercase, whitespace-free, and at most 20 characters. Use dashes or underscores for readability.

Method-level attribute

  • [MonitoredTest] replaces [Test] for monitored runs.

    The method name is used as test name in GUTS, and camelcase is split into words. For example, ShouldBeAtLeast300PixelsWide becomes Should be at least 300 pixels wide.
  • [MonitoredTest("Custom test name")] sets a custom report name (deprecated overload).
  • Use [MonitoredTest] together with [TestCase] for data-driven test methods.

Collection and submission flow

1. Class attribute initializes sender setup.
2. Each monitored factresult is captured.
3. Results accumulate per test class.
4. When all tests in a test class have completed, the results are sent to GUTS.
Results are only submitted when all tests in the class run. If only a subset runs, nothing is sent.
Results are accepted only when test-code hash matches a hash registered by a lector.
Hahes are automatically registered when a lector runs the test class.

If students modify test code, the backoffice rejects the submission.
